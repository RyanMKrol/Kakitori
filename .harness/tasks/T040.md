# T040 — Register .apkg document type (Open in Kakitori)

## Overview

Declare the .apkg document/UTI types in project.yml so "Open in Kakitori" appears in the share
sheet, and route incoming file URLs into the existing Home import path.

## Do

- **project.yml only** for the plist keys — the generated `Sources/Info.plist` and the `.xcodeproj`
  must NEVER be hand-edited (they are xcodegen output; regenerate with `xcodegen generate` after
  editing project.yml). Under the Kakitori target's `info.properties`, add
  (docs/04-content-and-data.md §2: "`UTType` registered for `.apkg`" + share-sheet "Open in Kakitori"):
  - `UTImportedTypeDeclarations` — one entry:
    - `UTTypeIdentifier: com.ryankrol.kakitori.apkg`
    - `UTTypeDescription: Anki Deck Package`
    - `UTTypeConformsTo: [public.data, public.zip-archive]`
    - `UTTypeTagSpecification: { public.filename-extension: [apkg] }`
  - `CFBundleDocumentTypes` — one entry:
    - `CFBundleTypeName: Anki Deck Package`
    - `LSItemContentTypes: [com.ryankrol.kakitori.apkg]`
    - `CFBundleTypeRole: Viewer`
    - `LSHandlerRank: Alternate`
- **`Sources/App/`** (the `@main` scene): attach
  `.onOpenURL { url in ImportCoordinator.shared.begin(url: url) }` to the root view, so an incoming
  .apkg lands in exactly the same import path as the Files-picker flow
  (`Sources/Features/Home/ImportCoordinator.swift` — its progress UI and specific-reason error
  alerts then apply unchanged). Single-window app: no `DocumentGroup`, no scene changes beyond the
  modifier (docs/05-architecture.md §6).

## Done when

- After `xcodegen generate`, both
  `plutil -extract CFBundleDocumentTypes xml1 -o - Sources/Info.plist` and
  `plutil -extract UTImportedTypeDeclarations xml1 -o - Sources/Info.plist` print the entries above
  (read-only verification — never edit that file).
- The `.onOpenURL` handler exists in `Sources/App/` and routes to `ImportCoordinator.shared.begin`.
- `./build_run.sh Kakitori-Sim -seedDemoData YES` builds and launches clean; Home renders unchanged.
