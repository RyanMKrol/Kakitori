# T046 — Mixed mode rotation + new-cards-always-trace-first

## Overview

A pure per-session mode resolver: Mixed rotates through the deck's available modes card by card,
and a card in state `.new` is ALWAYS presented in Trace for its first grading, whatever the session
mode (docs/03-srs-algorithm.md §6). Scheduling stays per card, never per mode.

## Do

1. Create `Sources/Scheduler/ModeResolver.swift` — a pure `struct ModeResolver` (Foundation only;
   references `PracticeMode` from `Sources/Features/Session/PracticeMode.swift` and `CardState`
   from `Sources/Models/CardSchedule.swift`, same target):
   - `init(sessionMode: PracticeMode, availableModes: [PracticeMode])` — `availableModes` is the
     deck's `ModeAvailability.deckModes(scripts:)` result (from `Sources/Scheduler/ModeAvailability.swift`)
     with `.mixed` removed; keep a private `rotationIndex = 0`.
   - `mutating func nextMode(cardState: CardState, qualifies: (PracticeMode) -> Bool) -> PracticeMode`:
     - `cardState == .new` → return `.trace` and do NOT advance `rotationIndex` (docs/03 §6: "New
       cards default to appearing in Trace regardless of the chosen mode for their first exposure").
       After first grading the state is no longer `.new`, so the card follows the session mode.
     - `sessionMode != .mixed` → return `sessionMode` if `qualifies(sessionMode)`, else `.trace`
       (safety net; queue filtering from the mode-availability task should already exclude these).
     - `sessionMode == .mixed` → starting at `rotationIndex`, scan `availableModes` circularly for
       the first mode that `qualifies`; return it and set `rotationIndex` just past it (so the next
       card gets the next mode — sequential rotation, prototype Mixed blurb "Rotate through all four
       modes"). If nothing qualifies, return `.trace`.
2. `Sources/Features/Session/SessionViewModel.swift`: hold one `ModeResolver` per session; each time
   a card is served, set the card's presented mode via
   `resolver.nextMode(cardState:qualifies:)` with `qualifies` backed by
   `ModeAvailability.cardQualifies` (audio via the VM's `AudioPlaying.isAvailable`). The presented
   mode drives everything already built: trace guides only for `.trace`, listen autoplay only for
   `.listen`, the per-mode prompt, and the top-bar mode label.

## Done when

`Tests/UnitTests/ModeResolverTests.swift` (XCTest) passes, asserting (all with a
qualifies-everything closure unless stated):

- Mixed, `availableModes: [.trace, .listen, .translate, .recall]`, five non-new cards → exactly
  `[.trace, .listen, .translate, .recall, .trace]` (deterministic wrap-around).
- Mixed with the same modes but `qualifies` returning false for `.listen` on the second card →
  the second card gets `.translate` and the third continues from `.recall` (unqualified mode skipped,
  rotation not stuck).
- Recall session, one card graded from new: first call with `.new` returns `.trace`; second call for
  the same card with `.learning` returns `.recall` — Trace forced exactly once.
- Mixed session, card in `.new` → `.trace`, and the following non-new card still gets
  `availableModes[0]` (a new card consumes no rotation step).
