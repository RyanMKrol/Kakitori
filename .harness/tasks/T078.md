## Overview
Follow-on visual/affordance polish for the Settings screen (SettingsView.swift). Two problems: (1) the page is plain — a hand-rolled VStack painting KakitoriTheme.paper on a KakitoriTheme.paper container, NOT using the rounded-card visual language (KakitoriTheme.surface + boxLine stroke) that Home/Summary/DeckCard use everywhere else; (2) the two numeric rows ("New cards per day", "Max reviews per day") are bare borderless right-aligned TextFields that look identical to static text, so users can't tell they're editable until they tap. This is the polish pass AFTER T068 (DONE), which deliberately replaced the old daily-limit Steppers with these clamped numeric TextFields — so this task must NOT reintroduce Steppers or touch the clamping/persistence logic; it restyles the presentation and adds a clear editable affordance around the existing input.

## Do
Restyle Sources/Features/Settings/SettingsView.swift so the settings sit inside the app's established rounded-card language: wrap the settings rows in a card using KakitoriTheme.surface as the background, clipped to a RoundedRectangle at KakitoriTheme.radiusMedium (16) or radiusLarge (20), with a 1pt KakitoriTheme.boxLine stroke overlay — mirroring how StatsRowView / DeckCardView / SummaryView build their cards (surface fill + boxLine stroke). Keep the four existing settings and their order: "New cards per day", "Max reviews per day", "Audio autoplay" toggle, "Show romaji" toggle. For the TWO numeric rows, give the TextField an OBVIOUS editable affordance so it no longer reads as static text — e.g. render it inside a small bordered/boxed pill (its own rounded background + subtle border/stroke or a faint inkFaint fill) that visually signals 'tap to edit', while keeping the field itself a clamped numeric TextField. Do NOT reintroduce a Stepper and do NOT change the clamping ranges or persistence: keep the @AppStorage keys exactly ("newCardsPerDay" default 10, "maxReviewsPerDay" default 100, "audioAutoplay" default true, "showRomaji" default true), keep the clamp bounds in commitNewCardsValue (1...50) and commitMaxReviewsValue (10...500) and their commit-on-blur / onSubmit behavior, and keep the .keyboardType(.numberPad). Preserve every accessibilityIdentifier unchanged: settings-new-per-day, settings-max-reviews, settings-autoplay, settings-romaji (the tappable affordance/pill must carry or wrap the same identifier on the editable field so UI tests still target it). Use KakitoriTheme tokens (surface, boxLine, inkFaint, accent, radiusMedium/radiusLarge) and kakitoriFont for text so the page matches the paper/ink aesthetic; keep the KakitoriTheme.paper screen background and the "Settings" navigationTitle. Scope stays within Sources/Features/Settings/** — do NOT edit Theme.swift; the existing tokens (surface, boxLine, inkFaint, radiusMedium, radiusLarge) are sufficient, so no new shared token is needed. Verify the result looks clean and consistent on BOTH the iPad (Kakitori-Sim) and the iPhone (Kakitori-Sim-Phone) per the standing dual-device rule — the card layout and the field affordance must render well on both.
