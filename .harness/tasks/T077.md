## Overview
The in-session top-middle progress bar and its "N done · M left" header text advance on every grade tap — INCLUDING "Again" — instead of on actual card completions. In SessionView.swift the derived `done` = viewModel.gradeCounts.values.reduce(0,+) (SessionView.swift:141-143), the header is Text("\(done) done · \(left) left") (SessionView.swift:107), and progressBar computes progress = done / (done + left) over a MOVING denominator (SessionView.swift:145-164). Because SessionQueue.markGraded re-appends a card whose post-grade state is .learning/.relearning (SessionQueue.swift:139-145), tapping "Again" both bumps gradeCounts AND re-queues the card, so the bar advances on a wrong answer even though the card is not finished, and the moving denominator lets it hover mid-way / over-run the initial count. The top-RIGHT chips already track completions correctly: the red "learn" chip (viewModel.learnCount → SessionQueue.learnCount, SessionQueue.swift:29-31) decrements only when a non-Again grade graduates a card out of learning. Desired: make the bar + header mirror that chip accounting — count cards COMPLETED, not attempts. 'Completed' means precisely: a graded card whose post-grade state is NOT .learning/.relearning — the exact event SessionQueue.markGraded uses to decide NOT to re-queue the card (a new/learning card graduating to .review, or a .review card leaving the session for good). Again keeps a card in learning (re-queued) → NOT completed. Distinct from per-deck daily-stats (a separate idea) and from T059 (empty-session finish). SessionQueue is a named high-value unit-test target with an injected clock (AppClock.now / now:). Verify the header text + bar on BOTH Kakitori-Sim (iPad) and Kakitori-Sim-Phone (iPhone).
