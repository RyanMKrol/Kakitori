## Overview
On Listen & Write (.listen) cards, audio never plays on a physical device — not on autoplay entry, not on the play button. All wiring already reaches AudioService correctly; the defect is inside AudioService itself, which never configures or activates an AVAudioSession, so device output is silently suppressed. Two secondary defects compound it: playDeckAudio swallows load errors via `try?`, and play(...) never TTS-falls-back when the audio file is present-but-missing.

## Do
In Sources/Audio/AudioService.swift only (the AudioService class behind the existing @MainActor AudioPlaying protocol): (1) Configure the shared AVAudioSession with category .playback and call setActive(true) once, lazily and idempotently, before the first playback — wrap in do/catch, log on failure, never crash; call this activation at the start of both playDeckAudio and speakTarget. (2) In playDeckAudio, replace the silent `guard let player = try? AVAudioPlayer(contentsOf: url) else { return }` with a do/catch that logs the underlying error; on failure (missing/unreadable/wrong-format file) fall back to speakTarget(target) so the user hears ja-JP TTS instead of nothing. This requires threading the card `target` into the missing-file path — pass it through so the fallback can speak it (e.g. add the target to the playDeckAudio signature, or route the fallback via the protocol so play(...) can retry with speech; keep the AudioPlaying protocol contract and FakeAudioPlayer's call-recording behavior intact and update the fake + existing AudioServiceTests to the new signature). (3) Do NOT change PromptPaneView, SessionViewModel, MediaStore, project.yml, or Info.plist. First-party AVFoundation only; @MainActor and Swift 6 strict concurrency must hold.

## Done when
AVAudioSession is configured with category .playback and setActive(true) is invoked before playback (grep for `AVAudioSession` + `.playback` + `setActive` in Sources/Audio/AudioService.swift returns matches, guarded by do/catch). playDeckAudio no longer uses `try?` for AVAudioPlayer(contentsOf:) — the error path logs and falls back to spoken TTS. A NEW unit test (in Tests/UnitTests/, using the AudioPlaying fake and/or an injectable file-existence seam) asserts that when audioFilename is non-empty but the underlying media file is missing/unreadable, playback routes to speakTarget(target) rather than producing no call at all; the three existing fallback-selection tests (nil / empty / present-filename) and testMediaFileURLResolution remain green after any signature change. All existing AudioServiceTests pass. Real on-device audibility is a by-hand check (not unit-testable) and is out of the automated bar.
