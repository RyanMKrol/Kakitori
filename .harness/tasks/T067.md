## Overview
Tapping 'Show answer' during a session reveals the answer with visible jank: a flash of the paper/background bleeds through the prompt-to-answer transition, and the canvas / bottom action area jumps as the full-width 'Show answer' button swaps to the four-button grade row. Fix the reveal so the answer updates in place cleanly — no background flash and no layout jump — on BOTH iPhone (compact) and iPad (regular) layouts, with Reduce Motion still respected.

## Do
The reveal is driven by viewModel.phase flipping .prompt -> .revealed and involves two regions that both animate:

1. PromptPaneView (Sources/Features/Session/PromptPaneView.swift): a ZStack { KakitoriTheme.paper; if phase == .prompt { promptView.transition(.opacity) } else { answerBlock.transition(.opacity) } } with .animation(.easeInOut(duration: 0.25), value: viewModel.phase). Because both the outgoing prompt and incoming answer are semi-transparent mid-crossfade, the paper/background shows through — the 'flash'. Fix so the background never bleeds through the transition: e.g. keep the prompt and answer content in the SAME opaque container so there is always a fully-opaque foreground layer over the paper during the swap (an asymmetric transition where the incoming layer is fully opaque before the outgoing fades, or a non-crossfading swap), rather than two independently-fading siblings over the paper. Preserve the existing prompt/answer copy and styling exactly — this is a transition change, not a redesign.

2. ActionRowView (Sources/Features/Session/ActionRowView.swift): swaps the full-width 'Show answer' button (showAnswerButton) for the grade row (gradeRow) via KakitoriTheme.emphasisTransition(reduceMotion:). The grade row can differ in height from the button — it stacks four buttons VERTICALLY once dynamicTypeSize.isAccessibilitySize — so the bottom action area changes height on reveal. In SessionView.swift's compactLayout the actionRow is pinned at the bottom with canvasPane at .frame(maxHeight: .infinity) above it (prompt band a fixed 280pt), so an actionRow height change reflows the canvas — the 'jump'. Fix so revealing does not shift the canvas or the action area: e.g. give the action area a stable height so the button and the (possibly taller, stacked) grade row occupy the same footprint, so the swap happens in place. Check the regularLayout path too (canvasPane maxHeight .infinity + actionRow in a VStack) so the fix holds on iPad as well.

Reduce Motion (@Environment(\.accessibilityReduceMotion)) MUST still be respected: under Reduce Motion use a plain, non-flashing crossfade with no scale/slide (KakitoriTheme.emphasisTransition already degrades to .opacity — keep that contract; if you touch Theme.swift, keep the reduceMotion branch motion-free). Keep the change scoped to the reveal transition/layout — do not alter the scheduler, viewModel phase logic, or card-to-card transition.

Because visual jank cannot be meaningfully unit-tested, acceptance is a manual/visual check: build and run on Kakitori-Sim (iPad) and Kakitori-Sim-Phone (iPhone), start a session, tap 'Show answer', and confirm a clean in-place reveal.
