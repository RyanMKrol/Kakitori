# T047 — Settings screen: 4 settings + persistence + privacy manifest

## Overview

A Settings screen reachable from Home with the four v1 settings (docs/02-product-spec.md §5), stored
via `@AppStorage`, consumed where the session is built, plus the UserDefaults privacy-manifest entry.

## Do

1. Create `Sources/Support/AppSettings.swift` — the single non-view read point for the four keys:
   `struct AppSettings { var defaults: UserDefaults = .standard }` with computed vars falling back
   to the documented defaults when a key is unset:
   - `newCardsPerDay: Int` — key `"newCardsPerDay"`, default `10`
   - `maxReviewsPerDay: Int` — key `"maxReviewsPerDay"`, default `100`
   - `audioAutoplay: Bool` — key `"audioAutoplay"`, default `true`
   - `showRomaji: Bool` — key `"showRomaji"`, default `true`
   Use `defaults.object(forKey:) as? Int/Bool ?? <default>` (a plain `integer(forKey:)` would read
   0, not 10). Include setters so tests can round-trip through the struct.
2. Create `Sources/Features/Settings/SettingsView.swift` — SwiftUI `Form`, `.navigationTitle("Settings")`,
   background `KakitoriTheme.paper` (`Sources/Support/Theme.swift`):
   - `Stepper` "New cards per day" — `@AppStorage("newCardsPerDay") private var newCardsPerDay = 10`,
     range `1...50`, `.accessibilityIdentifier("settings-new-per-day")`, current value visible in the row.
   - `Stepper` "Max reviews per day" — `@AppStorage("maxReviewsPerDay") ... = 100`, range `10...500`,
     `.accessibilityIdentifier("settings-max-reviews")`.
   - `Toggle` "Audio autoplay" — `@AppStorage("audioAutoplay") ... = true`, `.accessibilityIdentifier("settings-autoplay")`.
   - `Toggle` "Show romaji" — `@AppStorage("showRomaji") ... = true`, `.accessibilityIdentifier("settings-romaji")`.
3. Reach it from Home: in `Sources/Features/Home/HomeView.swift`, add a toolbar button — SF Symbol
   `gearshape`, `.accessibilityIdentifier("settings-button")` — that pushes `SettingsView` on the
   existing `NavigationStack`.
4. Wire the caps: the session-queue build (`SessionQueue` in `Sources/Scheduler/SessionQueue.swift`)
   already takes the new/day and reviews/day caps as parameters. At the call site where the session
   is constructed, replace literal `10` / `100` with `AppSettings().newCardsPerDay` /
   `AppSettings().maxReviewsPerDay` (docs/03-srs-algorithm.md §3 lists both as settings).
5. Privacy manifest (UserDefaults is a required-reason API): create
   `Sources/Support/PrivacyInfo.xcprivacy` — plist with `NSPrivacyTracking` false,
   `NSPrivacyTrackingDomains` and `NSPrivacyCollectedDataTypes` empty arrays, and
   `NSPrivacyAccessedAPITypes` containing one dict:
   `NSPrivacyAccessedAPIType` = `NSPrivacyAccessedAPICategoryUserDefaults`,
   `NSPrivacyAccessedAPITypeReasons` = `["CA92.1"]`. In `project.yml`, ensure the app target bundles
   it as a resource (add `Sources/Support/PrivacyInfo.xcprivacy` under the target's `resources` with
   `buildPhase: resources` if it is not already picked up), then verify the generated project builds.

  - After setting `25`, `250`, `false`, `false` through `AppSettings`, a second `AppSettings` over
    the same suite reads all four back.

  form with "New cards per day 10", "Max reviews per day 100", and both toggles on.
