# T055 — XCUITest smoke test: launch, seed, start a Trace session

## Overview

One end-to-end XCUITest smoke flow: seeded launch → open a deck → start a Trace session → reveal →
grade → session advances. Smoke, not coverage — exactly ONE test flow.

## Do

1. Create `Tests/UITests/SmokeFlowTests.swift` (XCTest/XCUITest; the project convention is XCTest,
   not Swift Testing). One test class `SmokeFlowTests`, one test method, this flow:
   - `let app = XCUIApplication()`; append launch arguments `["-seedDemoData", "YES"]`; `app.launch()`.
   - Wait for a deck row: query by identifier prefix, e.g.
     `app.descendants(matching: .any).matching(NSPredicate(format: "identifier BEGINSWITH 'deck-row-'")).firstMatch`
     with `waitForExistence(timeout: 15)` — do NOT hardcode a display name; the UI is bilingual and
     rows carry `deck-row-<name>` identifiers.
   - Tap the deck row → the deck setup sheet appears: wait for + tap `mode-trace`, then tap
     `start-writing`.
   - In the session: wait for `show-answer`, tap it.
   - Wait for + tap `grade-good`.
   - Assert the session advanced: `show-answer` exists again within 10s (the next card is served
     in its pre-reveal state, so the grade row is replaced by Show answer). Also assert
     `grade-good` no longer exists at that point.
2. House rules (binding): every wait is a bounded `waitForExistence(timeout:)` — no `sleep()`, no
   polling loops; every query is by `accessibilityIdentifier`, never display copy; no
   force-unwraps/`try!`.
3. Target wiring: ensure `project.yml` declares a UI-testing bundle (e.g. `KakitoriUITests`, type
   `bundle.ui-testing`, `target: Kakitori`, sources `Tests/UITests`) hooked into the `Kakitori`
   scheme's test action so the standard DoD command runs it. If such a target already exists, only
   add the new file. Never hand-edit the generated `.xcodeproj` — `project.yml` + `xcodegen generate`.

## Done when

- `xcodebuild test -project Kakitori.xcodeproj -scheme Kakitori -destination 'platform=iOS Simulator,name=Kakitori-Sim' -only-testing:KakitoriUITests/SmokeFlowTests CODE_SIGNING_ALLOWED=NO`
  passes on the dedicated `Kakitori-Sim` simulator (ensure it exists first: `./tools/loop_sim.sh`).
- The test performs exactly the steps above: seeded launch, deck-row wait+tap, `mode-trace`,
  `start-writing`, `show-answer` wait+tap, `grade-good` tap, then asserts `show-answer` reappears
  (session advanced) and `grade-good` is gone.
- The full local DoD suite (which runs `xcodebuild test` on the whole scheme) still passes with
  the new UITest target included.
