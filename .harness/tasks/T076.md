## Overview
Today the daily new/review caps and progress are GLOBAL: DailyStats (Sources/Models/DailyStats.swift) is keyed by `day: String` only (one row per calendar day, no deck reference); StatsRecorder writes into that single row; and HomeView passes the same global todayStats?.newIntroduced / reviewsDone into every DeckCardView (HomeView.swift:124-125, 237-238, 254-255). So introducing the daily new-card quota in ONE deck drives newIntroducedToday >= newPerDay for ALL decks — DailyAllowance.forDeck then returns newCount 0, and a fresh deck (learnCount 0, dueCount 0) shows total 0 → DeckCardView renders 'All caught up' (DeckCardView.swift:274-312) even though it holds 100+ unlearned cards. This is ideas #8 (the surface symptom) and #9 (the root framing) — the decks are ALREADY three separate Deck rows (Hiragana/Katakana/Kanji) split on import, and SM-2 is already per-note, so the ONLY gap is that the daily-progress/caps layer is global. Make daily progress and caps PER DECK so studying one deck never changes another deck's chips or caught-up state.

## Do
Make DailyStats per-deck. Add a stable deck key to the DailyStats @Model (the Deck already exposes `id: UUID` and a stable `sourceDeckName: String` — use one consistently as the key; a nil/absent key means legacy global rows). Because the app currently builds its ModelContainer from a bare `Schema([...])` with no VersionedSchema/migration plan (Sources/App/KakitoriApp.swift makeModelContainer), introduce the schema change so it does NOT crash on an existing on-device store: use an additive/lightweight SwiftData migration (a VersionedSchema pair + a SchemaMigrationPlan, or an additive optional property with a backfill) that preserves existing rows. Thread the studied deck through the write path: StatsRecorder.recordGrade / recordStudySeconds gain a deck (or deck key) parameter and fetch-or-create the per-(day, deck) row; SessionViewModel already holds the deck (it is constructed with `deck:`) so pass it at the call sites (SessionViewModel.swift:153, 197). Update DailyAllowance and the Home read path so each deck's newIntroducedToday / reviewsDoneToday come from that deck's OWN row, not the global one: HomeView.todayStats must resolve per-deck stats for each DeckCardView (HomeView deckList at :233-241 and :250-258, plus dueCount(for:) at :117-127), and DeckCardView keeps using DailyAllowance.forDeck with the per-deck figures. Keep the top-of-Home 'Written today' StatCard and the streak GLOBAL (aggregate DailyStats across decks) — StatsRowView (Sources/Features/Home/StatsRowView.swift) and DailyStats.currentStreak stay whole-app and must not regress; the streak's activeDays set is the union of all decks' active days. Do NOT re-split decks and do NOT touch SM-2 interval math.
