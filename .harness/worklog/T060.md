# T060 Worklog

## 2026-07-20

**Built:** A shared, pure `DailyAllowance` helper (`Sources/Scheduler/DailyAllowance.swift`) so the
Home banner, deck new/learning/due chips, and `DeckCardView.isAllCaughtUp` all report today's
capped allotment — the same numbers a session would actually serve — instead of the raw uncapped
backlog `TodayBannerView.calculateDueCards` and `DeckCardView`'s per-state counters used to walk
directly.

**Root cause:** `TodayBannerView.calculateDueCards` and `DeckCardView`'s `newCount`/`learningCount`/
`dueCount`/`isAllCaughtUp` counted every `.new`/`.learning`/`.relearning`/due-`.review` note with no
reference to `AppSettings.newCardsPerDay` / `.maxReviewsPerDay` or to how many were already
consumed today (`DailyStats.newIntroduced` / `.reviewsDone`). Those caps were applied only inside
`SessionQueue.build`, so the banner could read "215 characters" while a session would really only
serve ~20, and a deck could still show outstanding chips (never "all caught up") purely because the
uncapped backlog was nonzero even after the day's cap was hit.

**Implementation:**
- `Sources/Scheduler/DailyAllowance.swift` (new) — pure, `Foundation`-only `DailyAllowance` struct
  (`newCount`, `learnCount`, `dueCount`, `scripts` → `total`/`scriptCount`/`isAllCaughtUp`).
  `forNotes(_:now:endOfToday:newPerDay:maxReviewsPerDay:newIntroducedToday:reviewsDoneToday:)`
  mirrors `SessionQueue.build`'s cap logic exactly: due learning/relearning (`dueAt <= now`) is
  uncapped; new cards and due reviews (`dueAt <= endOfToday`, reviews sorted by `dueAt` ascending)
  are each capped by `remainingAllowance(cap:doneToday:) = max(0, cap - doneToday)`. `forDeck`/
  `forDecks` are `Deck`-level convenience wrappers; `aggregate` sums per-deck allowances and unions
  their `scripts` sets — this is the literal "aggregate matches the sum across decks" the spec asks
  for (each deck's new/review allowance is computed independently against the same global
  `newIntroducedToday`/`reviewsDoneToday`, matching what `SessionQueue.build` would serve if that
  specific deck were studied).
- `Sources/Support/AppClock.swift` — added `endOfToday(after:)`, extracted from the day-rollover math
  `SessionViewModel` already had privately, so `DailyAllowance` callers outside `Features/Session`
  (out of this task's scope) can compute the same boundary without duplicating it a third time.
- `Sources/Scheduler/SessionQueue.swift` — `build` now calls `DailyAllowance.remainingAllowance(cap:
  doneToday:)` for both the new and review allowances instead of inlining `max(0, cap - done)` twice,
  so the cap formula has one implementation. Left the interleave/ordering logic untouched per the
  spec ("do not change the queue's ordering/interleave").
- `Sources/Features/Home/TodayBannerView.swift` — added `@Query private var dailyStats: [DailyStats]`,
  `clock`/`settings` init params (defaulting to `.system`/`AppSettings()`), and a static
  `calculateAllowance(decks:dailyStats:now:clock:settings:)` that looks up today's `DailyStats` row
  and calls `DailyAllowance.forDecks`. Replaced the old `calculateDueCards` static + the `(Int, Int)`
  tuple UI with `allowance.total`/`allowance.scriptCount`, and the "All caught up" branch now keys
  off `allowance.total > 0` / `allowance.isAllCaughtUp`.
- `Sources/Features/Home/DeckCardView.swift` — added `endOfToday`/`newIntroducedToday`/
  `reviewsDoneToday`/`settings` init params (all defaulted so existing 3-arg call sites still
  compile), replaced the four separate counting properties with a single `allowance: DailyAllowance`
  computed property, and `newCount`/`learningCount`/`dueCount`/`isAllCaughtUp` now just read off it.
- `Sources/Features/Home/HomeView.swift` — added a `todayStats: DailyStats?` computed property
  (fetches the `DailyStats` row for `AppClock.system.adjustedDay(for: now)`, mirroring the existing
  `currentStreak` fetch pattern) and threads `newIntroducedToday`/`reviewsDoneToday` from it into
  both `DeckCardView` call sites (compact + regular grid). `dueCount(for:)` (feeding
  `DeckSetupSheet`'s "N cards due" line) now calls `DailyAllowance.forDeck(...).total` instead of the
  removed `calculateDueCards`, so that surface stays consistent with the same helper rather than
  keeping a second uncapped implementation alive. `TodayBannerView`'s own `@Query`-based `decks`/
  `dailyStats` needed no wiring from `HomeView` — unchanged existing pattern.
- Function-parameter-count: `DailyAllowance.forNotes`/`forDeck`/`forDecks` gave `newPerDay`/
  `maxReviewsPerDay` the same `SRSConstants.defaultNewPerDay`/`.defaultMaxReviewsPerDay` defaults
  `SessionQueue.build` already uses — SwiftLint's `function_parameter_count` rule doesn't count
  defaulted parameters, and this keeps the default consistent with the existing scheduler default
  rather than inventing a new one.
- Did not touch `Sources/Features/DeckSetup/DeckSetupSheet.swift` itself (out of scope) or add a
  "study beyond today's cap" bypass — explicitly out of scope per the spec's owner-confirmed note.
