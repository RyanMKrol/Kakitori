# T072 — Prune retired Tofugu decks left behind by the Foundations bundle swap

## 2026-07-20

**Summary:** `BundledDeckLoader.bundleVersion` was bumped 1→2 when the Foundations source deck
replaced the two Tofugu decks, but `load()` only ever imported/updated the currently-bundled decks —
it never deleted decks the bundle no longer produces. Any install that had already run version 1 kept
its two retired Tofugu decks (`sourceDeckName` "Tofugu Hiragana Anki Deck" / "Tofugu Katakana Anki
Deck") on disk alongside the three new Foundations decks, so Home showed 5 deck cards instead of 3.

**What I did:**
- `Sources/App/BundledDeckLoader.swift`: `load()` now derives the EXPECTED `sourceDeckName` set the
  current bundle produces by parsing each bundled `.apkg`'s root deck name via
  `ApkgImporter.parse(url:)` (the same source of truth the importer itself uses) and combining it
  with each `sectionTitles` key — i.e. `"<rootDeckName>::<section>"` — rather than hardcoding
  "Kakitori Foundations" as a literal string, so a future bundle swap stays correct. After a
  successful import pass, a new `pruneRetiredDecks(container:keeping:)` fetches every `Deck` and
  deletes any whose `sourceDeckName` isn't in the expected set, only on a load pass (never on the
  steady-state up-to-date path).
- `Sources/Models/Note.swift` / `Sources/Models/CardSchedule.swift`: `Note.schedule` had no explicit
  delete rule, so SwiftData's default nullify would have cleared `CardSchedule.note` and stranded the
  schedule row rather than deleting it when a pruned deck's notes were cascade-deleted. Moved the
  relationship's `@Relationship(deleteRule: .cascade, inverse: \CardSchedule.note)` annotation onto
  `Note.schedule` (matching the existing `Deck.sections` / `Section.notes` pattern in this codebase,
  where the parent side carries both the cascade rule and the inverse) and left `CardSchedule.note` a
  plain property. `pruneRetiredDecks` also explicitly deletes sectionless notes still linked directly
  via `Note.deck` (not reachable through the `Section` cascade) before deleting the deck itself.
- `pruneRetiredDecks` is `internal` (not `private`) so unit tests can drive it directly against a
  seeded in-memory container.

**Tests added (`Tests/UnitTests/BundledDeckLoaderPruneTests.swift`):**
- `testPruneRemovesRetiredDeckAndCascadesToNoteAndSchedule` — seeds a retired Tofugu deck (Section +
  Note + CardSchedule) alongside the 3 expected Foundations decks, runs `pruneRetiredDecks`, asserts
  exactly the 3 Foundations decks remain, the Tofugu note is gone, and no orphaned `CardSchedule` rows
  survive (asserts the remaining schedule count is exactly 3 — the 3 surviving decks' own schedules —
  and that none reference the deleted note).
- `testPruneIsNoOpWhenOnlyExpectedDecksExist` — seeds only the 3 expected decks, asserts nothing is
  deleted.
- `testLoadPrunesRetiredTofuguDecksAfterFoundationsSplit` — end-to-end: seeds the 2 retired Tofugu
  decks, runs the real `BundledDeckLoader.load()` against the real bundled Foundations `.apkg`, and
  asserts the store ends up with exactly the 3 Foundations `sourceDeckName`s.
- First run (before the cascade-rule fix) confirmed the CardSchedule-orphan bug for real: pruning
  deleted the Note but left its CardSchedule behind, failing the "no orphaned schedule" assertion,
  before I moved the delete rule and it passed.
