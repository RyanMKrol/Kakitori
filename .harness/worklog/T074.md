# T074 — Centre the guide-box grid and confine ink to it

## 2026-07-20

**Summary:** The writing canvas had two linked layout defects: (1) `GuideBoxRow`'s row `HStack` had
a trailing `Spacer()` that ate all leftover width, defeating the row's own
`.frame(maxWidth: .infinity, alignment: .center)` and jamming boxes to the leading edge (and, since
`GeometryReader` top-aligns its child by default, to the top too); (2) `WritingCanvas`'s
`PKCanvasView` filled the entire pane in `CanvasPaneView`'s `ZStack`, decoupled from the guide-box
geometry, so PencilKit accepted ink anywhere on the pane, not just inside the visible box(es).

**What I did:**
- Added `Sources/Features/Session/Canvas/GuideBoxGridGeometry.swift` — a pure, side-effect-free enum
  extracted from `GuideBoxRow`'s inline layout math: `rows(units:maxBoxesPerRow:)` (the row-wrapping
  logic) and `boxSize(forRowBoxCount:availableWidth:)` (the per-row box sizing), plus a new
  `gridSize(units:maxBoxesPerRow:availableWidth:)` that returns the bounding `CGSize` of the
  rendered, centred grid. This is the single source of truth both `GuideBoxRow` (which renders the
  grid) and `CanvasPaneView` (which now sizes the canvas to it) call, so they can't drift apart.
- `GuideBoxRow.swift`: removed the trailing `Spacer()` in the row `HStack` (fixes horizontal
  centring — the row now shrinks to fit and the `.frame(maxWidth: .infinity, alignment: .center)`
  actually centres it). Changed the content `VStack`'s frame from
  `.frame(maxWidth: .infinity, alignment: .center)` to
  `.frame(maxWidth: .infinity, maxHeight: .infinity, alignment: .center)` so the whole row-stack is
  also vertically centred within the `GeometryReader` (which top-leading-aligns its child by
  default) — without this, boxes stayed pinned to the top of the pane even after the horizontal fix.
  Replaced the private `layoutRows`/inline `boxSize` math with calls into the new
  `GuideBoxGridGeometry` (re-pairing rows with each unit's original global index for the
  `guide-box-<index>` accessibility identifiers).
- `CanvasPaneView.swift`: wrapped the canvas `ZStack` in a `GeometryReader`, computed
  `GuideBoxGridGeometry.gridSize(units:maxBoxesPerRow:availableWidth:)` for the pane's available
  width, and sized `WritingCanvas` to that `CGSize` (falling back to an unconstrained frame when the
  target is empty, e.g. no current note). Since the `ZStack` centres its children by default and
  `GuideBoxRow`'s content is now also centred within the same available space, the drawable
  `PKCanvasView` rect and the visible, centred guide-box region coincide — SwiftUI hit-testing means
  touches/Pencil input outside the canvas's (now box-sized) frame simply never reach the
  `PKCanvasView`, so ink cannot land outside the box(es). Extracted `maxBoxesPerRow` (4 compact / 6
  regular) into a computed property shared between `TraceGuideLayer` and the new geometry call, so
  the same per-orientation box count feeds both the rendering and sizing paths. No other behaviour
  (undo/clear, per-note clear, trace glyphs, ink colour resolution) was touched.

**Tests added:**
- `Tests/UnitTests/GuideBoxGridGeometryTests.swift` — unit tests on the new pure geometry: a single
  box sizes to `maxBoxSize`; a 3-box row sums box widths + inter-item spacing; an 8-unit target with
  `maxBoxesPerRow: 6` wraps into 2 rows (6 + 2) and the grid height stacks both row heights + row
  spacing; a narrow available width shrinks boxes below `maxBoxSize`; empty units produce `.zero`.
- `Tests/UnitTests/CanvasPaneRenderTests.swift` — added
  `testCanvasPaneCentersSingleCharacterBoxOnCompactWidth`, rendering the real `CanvasPaneView` at a

  iPad-sized (1194×834) render tests.
