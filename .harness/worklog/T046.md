# T046 — Mixed mode rotation + new-cards-always-trace-first

## 2026-07-18

Implemented `Sources/Scheduler/ModeResolver.swift`: a pure `struct ModeResolver` holding a
`sessionMode` and an `availableModes` list, with a mutable `rotationIndex` starting at 0. The
`nextMode(cardState:qualifies:)` method implements the three card-serving rules:

1. **New cards forced to Trace**: when `cardState == .new`, returns `.trace` without advancing
   `rotationIndex`, so the card "does not consume" a rotation step per docs/03-srs-algorithm.md §6.
2. **Non-mixed sessions**: when `sessionMode != .mixed`, returns `sessionMode` if the
   `qualifies` closure accepts it, otherwise `.trace` as a safety net (per spec: queue filtering
   should have already excluded unqualified cards).
3. **Mixed rotation**: when `sessionMode == .mixed`, scans `availableModes` circularly starting
   at `rotationIndex`, returns the first mode that `qualifies`, and advances `rotationIndex` just
   past it. If no mode qualifies, returns `.trace`. This gives the deterministic wrap-around
   behaviour ("Rotate through all four modes") documented in the mode's description.

Added `Tests/UnitTests/ModeResolverTests.swift` (XCTest) asserting all four task acceptance
criteria:

- Mixed mode with four modes and all-qualifying cards: `[.trace, .listen, .translate, .recall, .trace]`.
- Mixed mode skipping unqualified `.listen`: continues from next mode, rotation not stuck.
- Non-mixed (Recall) session: new card forced to `.trace`, same card in `.learning` returns `.recall`.
- New card in Mixed session: forced to `.trace`, following card gets `availableModes[0]` (no rotation step consumed).

Integrated `ModeResolver` into `Sources/Features/Session/SessionViewModel.swift`:

- Added `presentedMode: PracticeMode = .trace` field (used by views instead of the session
  `mode` for per-card control).
- Added `modeResolver: ModeResolver` field initialized in `init()` with `availableModes`
  filtered from `ModeAvailability.deckModes()` minus `.mixed` (per spec).
- Called `modeResolver.nextMode(cardState:qualifies:)` each time a card is served (in both
  the init path to `firstEntry` and in `grade()` when stepping to `nextEntry`), computing
  `qualifies` from `ModeAvailability.cardQualifies` with audio availability from the session's
  `audio` provider.
- Updated `autoplayOnCardEntry()` to check `presentedMode == .listen` instead of `mode == .listen`.

Swift 6 strict concurrency: wrapped the `qualifies` closure in `[weak self]` to avoid capturing
`self.audio` implicitly.
