# T010 — TargetSegmenter: writeable units + script classification

## 2026-07-17

**Summary:** Implemented pure-Foundation TargetSegmenter enum with segment() and classify() functions for splitting Japanese text into guide-box units and classifying script type by Unicode ranges.

**Implementation:**
- Created `Sources/Importer/TargetSegmenter.swift` with three public types:
  - `SegmentedUnit` enum: `.box(String)` for writeable characters, `.inline(String)` for punctuation
  - `TargetScript` enum: hiragana, katakana, kanji, mixed
  - `TargetSegmenter` enum with static functions
- `segment()` walks characters in order:
  - Checks punctuation set `。、！？・ー` first (sets drop `ー` from katakana block correctly)
  - Drops whitespace/newlines entirely
  - All other characters become `.box` units
  - Order preserved
- `classify()` scans Unicode scalar ranges of box units only:
  - U+3040–U+309F → hiragana
  - U+30A0–U+30FF → katakana
  - U+4E00–U+9FFF → kanji
  - Mixed script or presence of Latin/digits → mixed
  - Empty box list → mixed

**Tests:** Created `Tests/UnitTests/TargetSegmenterTests.swift` with 15 test cases covering:

- Chouon (ー) in katakana
- Long phrase with multiple punctuation marks
- Whitespace/newline dropping
- Edge cases (empty, punctuation-only, Latin/digits)
- Full assertion of `[SegmentedUnit]` array order via `XCTAssertEqual`

**Notes:**
- Refactored `classify()` to extract script detection into a private helper to keep cyclomatic complexity ≤10 for linter.
- Whitespace handling uses `||` (OR) not `,` (AND) — both `isWhitespace` and `isNewline` individually drop characters; combining with `,` would only match chars that are both simultaneously.
- Tuple return fixed by reducing to (Bool, Bool, Bool, Bool) instead of named fields; linter accepts max 2-member tuples, so used Set<TargetScript> to track scripts found instead.
