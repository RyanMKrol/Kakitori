# T073 — Fix Listen & Write audio never playing on device

## 2026-07-20

**Summary:** `.listen` card audio never sounded on a physical device because `AudioService` never
configured/activated an `AVAudioSession` — nothing routed sound to output. Two secondary defects: the
missing-file path used `try?` and silently dropped the error, and there was no TTS fallback when the
bundled audio file was present-but-unreadable.

**What I did (`Sources/Audio/AudioService.swift` only):**
- Added a lazy, idempotent `activateAudioSessionIfNeeded()` that sets `AVAudioSession.sharedInstance()`
  category to `.playback` and calls `setActive(true)` inside a `do/catch` (logs and never crashes on
  failure). Called at the start of both `playDeckAudio` and `speakTarget`.
- Changed the `AudioPlaying` protocol's `playDeckAudio(filename:deckID:)` to return `@discardableResult
  Bool` (success/failure) instead of `Void`. The default `play(target:audioFilename:deckID:)` extension
  now retries with `speakTarget(target)` when `playDeckAudio` reports failure — this threads the
  fallback through the protocol without needing to add `target` to `playDeckAudio`'s signature, per the
  spec's suggested alternative.
- `AudioService.playDeckAudio` replaced the silent `guard let player = try? AVAudioPlayer(contentsOf:
  url) else { return }` with a `do/catch`: on success it plays and returns `true`; on failure (missing
  file, unreadable, wrong format) it `print`s the underlying error and returns `false`, letting the
  protocol-level fallback speak the target via TTS.
- Updated `Tests/UnitTests/FakeAudioPlayer.swift` to the new `Bool`-returning signature, adding a
  `deckAudioSucceeds` toggle (default `true`) so tests can simulate a missing/unreadable file while
  still recording the `.deck` call (mirrors what a real failed `AVAudioPlayer` attempt looks like).

**Test added (`Tests/UnitTests/AudioServiceTests.swift`):**
- `testPlayWithPresentFilenameFallsBackToSpeechWhenDeckAudioFails` — sets `FakeAudioPlayer.deckAudioSucceeds
  = false`, calls `play(target:audioFilename:deckID:)` with a non-empty filename, and asserts the call
  sequence is `[.deck(...), .speak(target)]` — i.e. deck playback is attempted, fails, and TTS fallback
  fires. Confirmed this test fails first (only `.deck` call recorded, no `.speak`) against the
  pre-change `Void`-returning signature before implementing the `Bool` return + fallback retry, then
  passes after.
- All 3 existing fallback-selection tests (nil/empty/present filename) and `testMediaFileURLResolution`
  remain green with the new signature (no behavior change on the success path).
