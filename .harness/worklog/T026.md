# T026 — SessionQueue: queue building with daily caps + ordering

## 2026-07-17

### What I did
- Added `Sources/Scheduler/SessionQueue.swift` (`import Foundation` only, no SwiftData/SwiftUI,
  no `Date()`/`.now`, no RNG):
  - `QueueEntry { let id: UUID; var snapshot: ScheduleSnapshot }` (Equatable).
  - `SessionQueue` with `entries`, computed `newCount` / `learnCount` / `dueCount`, and a static
    `build(...)` following docs/03 §5:
    1. Learning/relearning with `dueAt != nil && dueAt <= now`, sorted by `dueAt` ascending.
    2. Reviews with `dueAt != nil && dueAt <= endOfToday`, sorted ascending, then
       `prefix(max(0, maxReviewsPerDay - reviewsDoneToday))`.
    3. New in input order, `prefix(max(0, newPerDay - newIntroducedToday))`.
    4. Deterministic fractional-position interleave: review i → `(i+1)/(R+1)`, new j → `(j+1)/(N+1)`,
       sorted ascending, reviews winning ties (same-type ties keep original order). Final order =
       learning block + merged list. Verified against both spec examples (R1 N1 R2 R3 N2 R4;
       R1 N1 R2 N2).
- Added `Tests/UnitTests/SessionQueueBuildTests.swift` (XCTest, snapshots by hand, no SM2Scheduler,
  `now = Date(timeIntervalSince1970: 1_000_000)`) covering every "## Done when" item: review cap vs
  reviewsDoneToday (5 earliest / 0), new cap vs newIntroducedToday (first 6 / 0), learning ordering
  from shuffled input, both interleave patterns, the 4 AM boundary (03:59 in / 04:01 out /
  `now + 1s` learning strictly excluded), and chip counts (2 new / 2 learn / 3 due).

### Note on `Sendable`
The spec signatures write `struct QueueEntry: Equatable, Sendable` and `struct SessionQueue: Sendable`.
Both types are internal structs whose stored members are all Sendable, so Swift *infers* Sendable —
and the DoD's SwiftFormat `redundantSendable` rule (CI-authoritative) rejects the explicit
annotation. I left the conformance implicit (documented with a comment); the types are Sendable as
the spec requires, and the DoD passes.
